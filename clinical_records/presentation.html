B-Tree Indexing in PostgreSQL

How It Works

  • Default index type in PostgreSQL
  • Balanced tree: root → internal → leaf pages
  • Leaf pages store (key, heap TID) pairs
  • Lookup cost: O(log n) vs O(n) seq scan

Planner Strategies

  • Index Scan few rows → pointer-by-pointer
  • Bitmap Index Scan many rows → batch heap read
  • Seq Scan when index won't help

Composite Index Benefit

Index on (patient_id, encounter_date):

WHERE patient_id = 50 ORDER BY encounter_date DESC
  • Left prefix → filters by patient
  • Right column → pre-sorted order
  • No extra Sort node in query plan

Diagnosis Lookup by Code

What the App Does

Clinician searches all diagnoses by ICD-style code (e.g. G43 – Migraine).

SELECT diagnosis_id, diagnosis_code, description FROM diagnoses WHERE diagnosis_code = 'G43'

What PostgreSQL Does Internally

Bitmap Heap Scan on diagnoses Recheck Cond: diagnosis_code = 'G43' Heap Blocks: exact=161 → Bitmap Index Scan on idx_diagnoses_code Execution Time: 0.69 ms
Seq Scan on diagnoses Filter: diagnosis_code = 'G43' Rows Removed by Filter: 18,901 Execution Time: 1.80 ms

Why it matters: Index reads 163 pages. Sequential scan reads every row. 2.6× faster — and the gap grows with data volume.

Patient Encounter History

What the App Does

Retrieve a patient's visit history, newest first.

SELECT encounter_id, encounter_date, chief_complaint FROM encounters WHERE patient_id = 50 ORDER BY encounter_date DESC LIMIT 10

What PostgreSQL Does Internally

Index Scan Backward using idx_encounters_patient_date Index Cond: patient_id = 50 Execution Time: 0.077 ms No Sort node in plan ✓

Why No Sort?

  • Composite index stores rows in (patient_id, encounter_date) order
  • Backward scan = descending date for free
  • PostgreSQL reads leaf pages in order — already sorted

Why it matters: Composite indexes eliminate Sort nodes — critical when paginating through large encounter histories.

MVCC in PostgreSQL

How It Works

  • Every row version stores xmin (created by) and xmax (deleted by)
  • UPDATE creates a new row version, marks old one expired
  • Each transaction sees a consistent snapshot
  • Readers never block writers; writers never block readers

READ COMMITTED (default)

  • Each statement sees the latest committed data
  • No dirty reads — uncommitted data invisible
  • New snapshot taken at each statement start

Row-Level Locking

  • SELECT FOR UPDATE — exclusive row lock
  • Concurrent writers wait; readers are not blocked
  • Lock released on COMMIT / ROLLBACK

Concurrent Update – MVCC

Scenario

Two clinical staff update the same treatment record simultaneously.


TimeSession 1Session 2
t1UPDATE outcome = 'Improved' (not committed)—
t2… waiting …READ → sees 'Improving' (old)
t3COMMIT ✓—
t4—READ → sees 'Improved' (new)

Internal Explanation

  • Session 1's update creates a new heap tuple with xmin = S1's xid
  • Old tuple's xmax = S1's xid (not yet committed)
  • Session 2 checks visibility: S1's xid not in committed set → returns old version
  • After COMMIT: S1's xid is committed → new version visible

Why it matters: No locking, no blocking reads. Critical for healthcare apps where reads must never stall.

Row-Level Locking

Scenario

A clinician locks a treatment record before updating to prevent conflicting writes.

-- Session 1 SELECT outcome FROM treatments WHERE treatment_id = 2 FOR UPDATE; -- (holds lock 4 seconds) UPDATE treatments SET outcome = 'Recovered' WHERE treatment_id = 2; COMMIT;

What Happens

  • Session 1 acquires exclusive row lock
  • Session 2 attempts UPDATE on same row
  • Session 2 BLOCKS — waits 3.5 s
  • Session 1 commits → lock released
  • Session 2 unblocks and succeeds

MVCC vs Locking

  • MVCC: read/write concurrency (readers don't block)
  • SELECT FOR UPDATE: write/write serialization
  • Together they cover all concurrency scenarios

Monthly Diagnosis Summary

What the App Does

Administrator generates monthly report of diagnosis counts.

SELECT DATE_TRUNC('month', e.encounter_date) AS month, d.diagnosis_code, COUNT(*) AS count FROM diagnoses d JOIN encounters e ON d.encounter_id = e.encounter_id GROUP BY 1, 2 ORDER BY month DESC, count DESC

Query Plan

Hash Join → Seq Scan on diagnoses (19,885 rows) → Hash / Seq Scan on encounters HashAggregate Group Key: month, diagnosis_code Execution Time: 25.9 ms

Why Seq Scan Here?

  • Query touches all rows in both tables
  • Index scan would add overhead with no benefit
  • Planner correctly chooses Seq Scan + Hash Join
  • Contrasts with Ops 1 & 2 (selective queries)

PostgreSQL vs MySQL vs MongoDB

Feature PostgreSQL MySQL (InnoDB) MongoDB
Primary index Heap + secondary B-tree Clustered B-tree (PK) B-tree on _id
Concurrency MVCC (no read locks) MVCC + gap locks Document-level locking
Composite index Skip sort, prefix scan Similar support Compound indexes
EXPLAIN EXPLAIN ANALYZE + BUFFERS EXPLAIN FORMAT=JSON explain("executionStats")
Schema Strict relational Strict relational Flexible document

Key difference: PostgreSQL's heap storage separates table data from indexes — any column can be efficiently indexed without reorganizing the table. MySQL's clustered index means secondary index lookups always double-fetch through the PK.

Conclusion

Summary & Lessons Learned

B-Tree Indexing

  • 2.6× faster lookup with index
  • Bitmap scan for high-selectivity queries
  • Composite index eliminates Sort node
  • Planner chooses Seq Scan for full-table aggregations

MVCC

  • No dirty reads under READ COMMITTED
  • Readers and writers don't block each other
  • Row versioning via xmin/xmax in heap
  • SELECT FOR UPDATE for write serialization

Key Insight

  • Every app operation maps to an internal DB behavior
  • Index design directly affects query plan
  • Dataset size determines whether index kicks in
  • MVCC is why PostgreSQL excels in concurrent workloads
